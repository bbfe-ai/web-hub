# WebHub 轻量化私域团队版设计文档

> 版本：v1.0  
> 日期：2026-05-27  
> 定位：小团队（5-30 人）的"共享工具桌面"  
> 原则：开箱即用、零运维、拒绝企业级膨胀

---

## 一、产品定位

**一句话定义**：团队的浏览器主页 + 共享书签墙 + 工具备忘录。

**对标产品**：Heimdall、Homarr、Notion 工具页。  
**差异化**：比通用仪表盘更聚焦"团队共享"这一件事，比 Notion 工具页更轻、更快、零订阅。

### 核心使用场景

- **新人入职第一天**：打开它，团队所有内部系统、文档、AI Agent 一目了然，省掉一上午找人问链接
- **老员工日常**：不再依赖浏览器书签（同步麻烦、别人看不到、链接挂了没人知道）
- **团队 Leader**：不用再发"工具清单"文档，工具更新即同步给所有人
- **跨团队协作**：开会时一个 URL 直达，不用截图发链接

---

## 二、设计原则

### 五条铁律

1. **零账号也能用**  
   默认匿名访问，可选简单口令保护（一个团队一个密码，进得来就是自己人）

2. **单容器单文件**  
   保持 SQLite，`docker run` 一行起，数据一个文件备份

3. **没有角色**  
   所有人都能加、能改、能删——靠"操作历史"而不是"权限"兜底

4. **不做审批流、不做工单**  
   内部信任，编辑即生效

5. **少即是多**  
   每个功能上线前问一句："30 人团队真的需要吗？"

---

## 三、当前系统诊断

### 优点

- 极简栈（Express + SQLite + 原生前端），部署成本低，适合内网快速落地
- 截图 + 卡片视觉化，比纯链接列表友好
- 分类 + 搜索 + 访问计数雏形已具备

### 主要矛盾

**不在功能多少，而在"团队味"不够**：

- 看起来像个人书签收藏夹，没有"这是我们团队的"感觉
- 卡片信息密度低，6 个项目就把首屏占满
- 没有"谁加的"、"什么时候加的"、"最近谁在用"的协作信号
- 截图很美但价值有限——真正进来用的人不需要看截图，需要快速点击

## 四、功能规划（按价值/成本排序）

### P0：让"团队感"立刻浮现（1-2 天工作量）

| # | 功能 | 描述 | 实现要点 |
|---|------|------|---------|
| 1 | 团队问候横幅 | 右上角显示"XX 团队工具墙 · 共 N 个工具 · 今天 N 次访问" | 配置文件 `team.json` + 首页头部组件 |
| 2 | 谁加的/谁改的 | 每张卡片显示昵称（首次访问自选/手输身份，localStorage 记忆） | 数据库新增 `created_by`、`updated_by` 字段 |
| 3 | 最近热门 Top 3 | 首屏顶部横向卡片，基于 `click_count` 排序 | 复用现有点击计数，前端单独区块 |
| 4 | 新上架标签 | 7 天内新增的卡片角标显示 NEW | 基于 `created_at` 判断，纯前端 |
| 5 | 键盘快捷键 | `/` 聚焦搜索、`G` 切换网格/列表、`数字键 1-9` 直达分类 | 全局 keydown 监听 |

### P1：让"找得到"更顺手（3-5 天）

| # | 功能 | 描述 | 实现要点 |
|---|------|------|---------|
| 1 | 标签替代单分类 | 一个工具可同时是"AI"+"内部"+"实验中" | 新增 `tags` 表（多对多），保留 `category` 兼容 |
| 2 | 全文搜索 | 名称 + URL + 描述 + 标签 + 备注，支持拼音首字母 | SQLite FTS5 + 前端 pinyin-pro 库 |
| 3 | 个人化"我常用" | localStorage 记录本人点击 Top 5，固定顶部 | 纯前端，无需登录 |
| 4 | 批量添加 | 粘贴 URL 列表自动解析 title/favicon 入库 | 后端 `/api/projects/batch`，并发抓取 metadata |
| 5 | 卡片紧凑视图 | 一行 4-6 个 favicon + 名字，给重度用户 | 视图切换按钮，CSS Grid 调整 |

### P2：让"持续可用"自动化（5-7 天）

| # | 功能 | 描述 | 实现要点 |
|---|------|------|---------|
| 1 | 链接探活 | 后台每天 ping 一次，挂掉的卡片右上角红点 | `node-cron` + axios HEAD 请求 |
| 2 | 自动截图刷新 | 每周用 Playwright 重拍一次截图 | `node-cron` 调度，与现有截图能力复用 |
| 3 | 自动 favicon 抓取 | 加一个 URL 就有图标 | 解析 `<link rel="icon">` 或回退 `/favicon.ico` |
| 4 | 备注/小贴士 | 每张卡片下加一行"踩坑提示"（如"需先连 VPN"） | 新增 `tips` 字段（短文本） |
| 5 | 二维码分享 | 手机扫码直达，开会演示场景 | `qrcode` 库前端生成 |

### P3：私域加分项（可选）

| # | 功能 | 描述 |
|---|------|------|
| 1 | 每日一句 | 首页随机显示团队某人写的"今日推荐这个工具因为..." |
| 2 | 简单评论 | 每个工具下最多 5 条短评（500 字内），便利贴风格 |
| 3 | 导出导入 | JSON 一键导出/导入，迁移零成本 |
| 4 | 小屏适配 | 手机访问也舒服，团队群里发链接就能用 |

## 五、刻意不做的事（避免膨胀）

| 别人会建议的功能 | 为什么不做 |
|----------------|----------|
| SSO / LDAP | 30 人以下不需要，密码保护就够 |
| 角色权限 | 团队内部信任，没人故意搞破坏 |
| 审计日志 | 编辑历史显示最后修改人即可，不需要合规级日志 |
| 审批流 | 想加就加，错了改回来，比走流程快 |
| Redis / Postgres | SQLite 在 100 人以下场景完全够 |
| 多团队隔离 | 每个团队跑一个实例，docker 起一个不费事 |
| 通知中心 | 30 人面对面或群里说一声，比站内信高效 |
| 工单系统 | 不是工具门户的本职 |

## 六、技术栈与架构

### 保持不变

- **后端**：Express + better-sqlite3
- **前端**：原生 HTML/CSS/JS，无框架
- **部署**：单 Docker 容器，挂载一个 `data/` 目录

### 新增依赖（最少化）

| 依赖 | 用途 | 引入阶段 |
|------|------|---------|
| `node-cron` | 后台定时任务（探活、截图刷新） | P2 |
| `pinyin-pro`（前端 CDN） | 中文拼音搜索 | P1 |
| `qrcode`（前端 CDN） | 二维码生成 | P2 |

### 数据库 schema 增量

```sql
-- P0: 协作信号
ALTER TABLE projects ADD COLUMN created_by TEXT;
ALTER TABLE projects ADD COLUMN updated_by TEXT;

-- P1: 标签系统
CREATE TABLE tags (
  id INTEGER PRIMARY KEY,
  name TEXT UNIQUE NOT NULL
);
CREATE TABLE project_tags (
  project_id INTEGER,
  tag_id INTEGER,
  PRIMARY KEY (project_id, tag_id)
);

-- P1: 全文搜索（FTS5 虚表）
CREATE VIRTUAL TABLE projects_fts USING fts5(
  name, url, description, tips, tags,
  content='projects', content_rowid='id'
);

-- P2: 健康状态 & 备注
ALTER TABLE projects ADD COLUMN tips TEXT;
ALTER TABLE projects ADD COLUMN health_status TEXT DEFAULT 'unknown';
ALTER TABLE projects ADD COLUMN last_checked_at DATETIME;
```

### 配置文件

新增 `config/team.json`（可选，不存在时使用默认值）：

```json
{
  "teamName": "我的团队",
  "welcomeMessage": "欢迎使用团队工具墙",
  "password": "",
  "showWhoAdded": true,
  "compactViewByDefault": false
}
```

## 七、视觉与交互升级

### 首屏密度

- 卡片网格：当前一行 2 张 → 一行 3-4 张（响应式断点调整）
- 顶部新增"团队最近在看"：横向滚动展示近 24 小时点击的工具

### 视图切换

- 默认网格视图（截图卡片）
- 新增列表视图：表格形式，名字 + 标签 + 最近访问 + 操作（给老用户提速）
- 新增紧凑视图：favicon + 名字一行多个（重度用户）

### 交互细节

- 删除按钮加二次确认（避免误删别人加的）
- 卡片悬停显示"最后修改人 + 修改时间"
- 搜索框支持快捷键 `/` 聚焦
- 分类侧栏支持 `数字键 1-9` 直达

## 八、产品成长路径

### 第一阶段：自用打磨（5 人核心团队）
**目标**：把"团队感"和"找得到"两件事做到位  
**对应**：P0 全部 + P1 的标签和搜索

### 第二阶段：部门扩散（20-30 人）
**目标**：从可用到顺手，降低维护成本  
**对应**：P1 全部 + P2 的探活和批量导入

### 第三阶段：跨团队复制
**目标**：被多个团队各自部署  
**对应**：一键迁移/克隆模板 + 导出导入 + 配置化品牌

## 九、实施路线图

```
Week 1  ─ P0: 团队感首屏 + 添加人/修改人 + 最近热门 + NEW 标签 + 快捷键
Week 2  ─ P1.1-P1.2: 标签系统 + 全文搜索（含拼音）
Week 3  ─ P1.3-P1.5: 我常用 + 批量添加 + 紧凑视图
Week 4  ─ P2.1-P2.3: 链接探活 + 自动截图 + favicon 抓取
Week 5  ─ P2.4-P2.5 + P3: 备注 + 二维码 + 评论/导出/导入
```

每个阶段完成即可独立发布，无需等全部完工。

## 十、成功度量

不做复杂的数据看板，关注三个核心指标即可：

1. **DAU/团队人数**：> 60% 说明真的成了"主页"
2. **每周新增条目数**：> 3 说明内容在持续沉淀
3. **链接挂掉响应时间**：从坏到修复 < 3 天，说明探活和团队反馈机制有效

## 附录：与企业版的取舍对照

| 维度 | 企业版（已弃） | 轻量私域版（当前） |
|------|--------------|------------------|
| 用户体系 | SSO + RBAC | 匿名 + 可选密码 |
| 数据存储 | PostgreSQL + Redis | SQLite 单文件 |
| 审批流程 | 申请-审核-上架 | 直接编辑生效 |
| 部署形态 | docker-compose 多服务 | 单容器 |
| 元数据 | 20+ 字段（owner/SLA/密级…） | 8 字段以内 |
| 目标规模 | 100-1000 人 | 5-30 人 |
| 维护成本 | 需专人运维 | 零运维 |

**核心理念**：把企业级方案的"完备"换成了私域场景的"顺手"。
